End-to-end software delivery, as a set of agents

From a market problem to production -- and back again

A toolkit of 52 specialized agents that carry a software project across its whole life: discover the need, write the specs, break work into PBIs, run the sprint, write & review code, test it, ship it, operate it, and learn from it. Every stage hands off cleanly, traces by stable IDs, and is gated for quality -- engineered as a chain of fast feedback loops.

01 . Executive summary

In one line: a standardized, AI-ready operating system for software delivery -- 52 specialist agents that turn an idea into shipped, operated software with consistent quality gates and full traceability, usable by a classic team or a lean AI-augmented pod.

Read top to bottom -- this walks the full story in order: why delivery is fast (02, the core idea) -> the lifecycle (03) -> who runs it (04) -> the minimal pod that runs the loops (05) -> the toolset (06) -> a unit of work end-to-end (07) -> a developer's & tester's day (08) -> where artifacts live (09) -> how the team operates it (10) -> the connecting thread (11) -> the ask (12).

The business case

  • Faster, safer delivery -- every stage is a fast, gated feedback loop; defects are caught at the cheapest point.
  • Lower key-person risk -- knowledge lives in standard artifacts, not in heads.
  • Audit-ready governance -- traceability + RAID + go/no-go on by default.
  • AI-leverage -- the specs & gates are exactly the guardrails autonomous AI coding needs.

What it is (engineer's view)

  • 52 role-specialist agents + 50 templates + 22 skills + 13 slash commands across the full SDLC.
  • An automation layer that auto-builds the agent index and enforces RTM, PBI-orphan & ADR-reference integrity (Python hooks).
  • Portable -- one source emits native bundles for Claude Code, Cursor, Copilot & Windsurf (no lock-in).
  • Stack-agnostic core + FE/BE/API/data depth.
  • One system of record per artifact: wiki/repo for docs, Azure Boards for PBIs, linked by IDs.
  • Maps 1:1 onto Azure DevOps (Boards, Repos, Pipelines, Test Plans, Wiki).

Honest status

  • Coverage & consistency: strong (audited, no orphans).
  • Prompt-engineering quality: clean (no over-triggering).
  • Validation: early signal, full pilot pending -- an interim proxy run on live code (PILOT-READOUT.md) showed the gates catch real defects; the human-team pilot is the next proof.
  • Critic's rating: ~7.8 / 10 -- excellent blueprint, prove it with one full pilot.
The ask / next step: approve a 4-6 week pilot on one real project (or the lean AI-native pod) to validate output quality and measure cycle-time and defect-escape impact -- that's what moves this from "excellent design" to "proven practice."

The document spine -- what each abbreviation means and answers

Read top-down: each document refines the one above, from the business problem to build-ready detail. Colour marks altitude -- business, product, technical.

Order of execution (full end-to-end) -- top to bottom; each step is reviewed & gated before the next. The spec stages run once, the deliver stage repeats every sprint, and governance runs the whole time.

DefineMRD -> BRD -> PRD -> FRD
DesignSRS -> SDD -> TSD (+ ADRs . URS* if regulated)
Deliver  each sprintbacklog / PBIs -> estimate -> sprint plan -> code -> PR review -> CI/CD -> QA -> release (go/no-go)
Operatemonitor -> incident -> postmortem    feeds new PBIs back into Deliver
ThroughoutRAID . status reports . coding standards . AI-orchestration -- all tied together by the RTM

It's sequential with feedback loops, not rigid waterfall -- a finding can send you back up (e.g. SDD -> PRD), and Operate feeds new work into Deliver. You rarely run all of it: lean / AI-native teams collapse the spec stages (a 1-pager PRD + a lean RFC); doc-strategy-advisor picks the right subset.

Regulated work adds URS (User Requirements Specification) as a design input; significant choices are logged as ADRs (Architecture Decision Records). After the spec spine comes delivery: backlog -> sprint -> code -> PR review (Pull Request) CI/CD (Continuous Integration / Continuous Delivery) QA (Quality Assurance) ' release -> operate.

Governance (RAID -- Risks/Assumptions/Issues/Decisions, plus status reports) and inner-loop standards (onboarding, coding standards) run continuously alongside. Everything is tied together by one RTM -- Requirements Traceability Matrix -- so every requirement traces down to the test that proves it.

How this connects to Loopwright (section 02)

This lifecycle is loop engineering made concrete -- the same feedback principle at three nested scales. Each phase boundary is a gate; the arrows back up are the loops.

Loop (from section 02)Where it lives in this lifecycleCadence
Inner loopA developer inside Deliver: code -> build -> test -> debugseconds-min
Outer loopThe sprint & integration cycle: backlog -> PR review -> CI/CD -> QA -> releaseminutes-days
Outermost loopThe whole lifecycle: Define -> Design -> Deliver -> Operate back to backlogweeks-release

The feedback arrows above (SDD -> PRD re-scope, Operate new PBIs) and the "Throughout" governance row are exactly the loops section 02 says to keep fast and clear. So section 02 is the principle; section 03 is the principle instantiated.

Right-sizing the team

05 . Minimal pods -- the small teams that run the loops

Loop engineering's rule -- fewer hands per change = tighter loops -- points to a small, cross-functional pod that owns a slice end to end and runs its own inner + outer loop. Scale a product by adding pods, not by enlarging one team.

The AI-native pod (4-5 people)

The 2026 minimal viable team -- delivers what a 10-person team did a few years ago, with AI agents handling most implementation under human guardrails.

  • AI Orchestrator (senior eng) -- sets guardrails, delegates to AI agents, validates output
  • Product Lead (AI-literate) -- PRD, backlog priority, acceptance
  • Data / Integration specialist -- data quality & integrations
  • QA / Validation lead -- validation strategy & sign-off
  • + AI agents -- most implementation, under ai-orchestration guardrails

How a pod maps to the loops

  • Inner loop lives inside the pod -- its devs / AI edit -> build -> test in seconds.
  • Outer loop is the pod's own PR -> CI -> QA -> release -- few handoffs, so it stays fast.
  • Own a vertical slice (a feature or service) so the loops stay inside the pod; cross-pod handoffs are what slow the outer loop.
  • Scale = more pods running their loops in parallel -- not one bigger team with longer loops.

A pod's sprint -- the delivery loop in practice

Each sprint, the pod runs the whole outer loop on its own slice and ships -- no waiting on other teams. Same cadence as the 11-person team, just fewer hands per lap.

PlanProduct Lead sets the sprint goal & pulls Ready PBIs (sprint-planner . estimation-facilitator)
BuildAI Orchestrator delegates to AI agents; the pod codes & unit-tests (inner loop . ai-orchestration . coding-standards)
ReviewOrchestrator / peers review PRs against the bar (code-reviewer)
TestQA / Validation lead runs tests, logs bugs, signs off (test-plan-writer . qa-tester)
ShipPod releases its slice via CI/CD with go/no-go (ci-cd-pipeline . release-manager)
Improve Retro tunes the loop; Operate feedback & new PBIs return to the backlog (retrospective . incident-postmortem)

The pod owns this loop end to end for its slice, so the sprint completes inside the pod -- that's what keeps the delivery loop fast. Larger products run several pods, each on this same sprint loop in parallel.

08 . A day in the loop -- developer & tester

The other sections are the management view -- plans, gates, governance. This is the same work from the IC's seat: the inner loop where ~80% of the day actually goes. Same agents, ground level.

Two coupled loops: the developer's fast inner loop and the tester's QA loop run in parallel all sprint -- merge -> staging feeds the tester, bug -- fix bounces between them, and QA sign-off exits to the release. The management sections exist to keep both loops fast and unblocked.

Sprint days -> the end goal

Days 2-9 Dev: code -> PR -> CI

Day 9code freeze . QA full regression . last fixes

QA shifts left: the tester is in planning & refinement -- reviewing each PBI's acceptance criteria for testability and planning the tests (testability is a Definition-of-Ready check), not just testing at the end. 2-week sprint; the end goal is a shippable increment accepted by day 10 -- whatever isn't Done returns to the backlog, not the deadline.

Step by step, from each seat

A developer's day

  1. Pull a Ready story from the sprint board board / backlog
  2. Read its acceptance criteria + the rules; branch feature/PBI- dev-onboarding . coding-standards
  3. Work the inner loop: edit -> build -> unit-test -> debug (seconds) inner loop
  4. Open a PR with the template; CI runs lint/test/scan ci-cd-pipeline
  5. Address review feedback, iterate until approved code-reviewer
  6. Merge on green + approval -> deploys to staging DoD gate
  7. Raise an ADR if a real decision came up; pick up the next story adr-writer

If a spec is unclear, push back -- don't guess (it flows back to frd/prd).

A tester's day

  1. In refinement: review acceptance criteria for testability & plan the cases test-plan-writer
  2. Take a build on staging that meets the DoD + the PR's "how to test-- dev'QA handoff
  3. Execute the planned cases for the story test-plan-writer
  4. Do exploratory testing around it -- find what the cases miss qa-tester
  5. Log reproducible bugs: steps, expected/actual, severity vs priority bug report
  6. Dev fixes -> verify against the original repro + regress around it retest
  7. Sign off against exit criteria QA sign-off
  8. Verdict feeds the release go/no-go ' release-manager

A bug nobody can reproduce is noise; keep severity (impact) and priority (urgency) separate.

Both run inside the sprint loop (section 05). The developer lives in the fast inner loop; the tester guards the outer loop before customers see it. The management sections exist to keep these two loops fast and unblocked -- not the other way around.

The right model

  • Project/feature docs -> wiki/repo (written once, evolve).
  • PBIs -> the tracker; story + acceptance criteria + links live on the work item.
  • Everything connected by RTM IDs (BRD'PRD'FRD'SR'PBI'test).
  • A design doc/RFC covers a complex feature (many PBIs), not one-per-story.

A file per PBI (pbi/PBI-123.md)"?

  • Usually no -- it duplicates the tracker -> two sources of truth -> drift.
  • OK only when: a complex feature needs a design doc (linked from the item); you're git-native (the file is the tracker); or as a staging file the backlog-manager emits, then imported to Azure.
  • If you're updating the same fact in two places, the architecture is wrong.

The other half of IA (not just "what lives where")

  • Access & ownership -- least-privilege edit rights; exactly one named owner per artifact.
  • Versioning & history -- git history (repo), page history (wiki), revision history (work items); never overwrite silently.
  • Retention & lifecycle -- supersede ADRs (don't edit), archive stale docs, close/triage old items.
  • Discoverability -- a single docs landing page indexes where each thing lives, so people find the source of truth instead of recreating it. The wiki-curator agent builds and maintains this front door.

The engine underneath

02 . Loop engineering -- why delivery speed is loop speed

Faster feedback at every step = faster, safer shipping.

Software speed is loop speed. Delivery is nested feedback loops -- act -> get feedback -> correct. The whole toolkit is, in effect, a set of loops engineered to return feedback fast and clearly.

Two loops

Inner loop -- one developer editing, building, testing, debugging in seconds. Where flow lives. Maximise time here and make each lap fast.
Actors: the individual developer (frontend / backend) -- solo.

Outer loop -- the team integrating, reviewing, shipping, operating over minutes to days. Minimise how often and how long it interrupts the developer.
Actors: reviewers (leads / seniors), DevOps / SRE (CI/CD & ops), QA / tester, release manager, and on-call -- plus the developer responding.

Feedback must be fast and clear -- DORA's 2025 data found "clear feedback on the outcome of my tasks-- is the platform capability most correlated with good developer experience.

DORA: deploy freq . lead time . change-fail . MTTR Flow: cycle time SPACE / DevEx: flow . cognitive load . feedback

Why it matters more in 2026

The case for

  • AI flipped the bottleneck. When code is generated in seconds, the constraint moves from writing to reviewing, integrating & verifying -- i.e. the loops. Faster authoring with slow loops just piles up unreviewed risk.
  • Volume & drift. More AI-generated change per day means more to validate; only fast, clear loops keep quality from collapsing.
  • Platform engineering is near-universal (~90% of orgs run an internal developer platform) precisely to shorten outer-loop waits via self-service.
  • Cost-of-delay compounds. A bug caught at the keyboard costs seconds; in production, a postmortem. Loops decide where you pay.

Where teams get it wrong (so we don't)

  • Metric gaming (Goodhart's law). Optimise velocity or DORA numbers and teams inflate points or ship trivial PRs to juice deploy frequency. The dashboard goes green; value doesn't.
  • Local-optimisation trap. Teams shave 5 min off CI while the real delay is two-day decision latency or vague requirements. The cheap-to-measure loop gets tuned; the expensive one is ignored.
  • Tooling & platform overhead. An IDP is itself a system to build and maintain -- a new loop. For an 11-person team, heavy platform investment can cost more than it saves.
  • Speed without clarity = churn. Faster loops that return cryptic failures just let you make mistakes faster. The clarity half is harder and often skipped.
  • AI's new bottleneck. Naive "ship faster-- with AI can increase rework -- the verification loop, not the keystroke, is now the constraint.
The real problem isn't whether loops matter -- it's which loop you optimise. Teams tune the measurable inner loop (build/CI time) and neglect the expensive, fuzzy loops: requirement clarity, decision latency, and review quality. And in 2026, AI throughput makes the verification loop the true constraint. So loop engineering matters more than ever -- but only if you instrument the whole chain (this toolkit's gates and RTM do exactly that) rather than chasing one green number.
