Wheelwright - how it works, end to end

Software, built like a wheel.

A wheelwright builds the wheel that keeps turning. Wheelwright is a crew of 52 specialist AI agents -- one for each step of building software, from the market question down to shipped, monitored code.

You don't open an app or learn 52 commands. You describe the work; the right specialist picks it up, does it, and hands the result to the next one along.

Say this to a teammate

The 30-second version

"It's a team of ~52 AI agents, each owning one step of building software -- market research, requirements, design, backlog, code review, releases. You don't learn 52 commands. You just describe what you want -- 'write a PRD for our login feature' -- and the right agent picks it up, does it, and hands off to the next."

-- the whole idea, in one breath

How you actually use it

Three moves, that's it

There's nothing to launch. The fleet loads with your project and waits for plain English. Every job is the same shape:

4 - Live spec queries -- MCP, optional

The bundled server exposes 18 tools in two groups. RTM / catalog / conformance (12): trace/coverage (trace_requirement, rtm_coverage, funnel_status), catalog (list_requirements, find_artifact, search_specs...), conformance (check_conformance), and authoring (next_req_id, scaffold_project). Agent context memory (6): save_context / recall_context / list_contexts / get_context / delete_context / sync_contexts -- a token-efficient handoff layer so agents retrieve prior decisions in one call instead of re-reading upstream docs. The only prerequisite is uv -- .mcp.json launches the server with uv run --with "mcp[cli]", which provisions the MCP SDK and a Python interpreter on first run (no manual pip install).

Information architecture

Where every artifact lives

One rule above all: one system of record per kind of thing -- never duplicate, link by ID. If you're updating the same fact in two places, the layout is wrong. The deliverables are visible and content-named; the tool's own state is hidden and disposable.

One home per kind of thing

ArtifactLives inWhy
Durable docs -- MRD, BRD, PRD, SRS, SDD, TSD, URSdocs/ in the repo (or a wiki)Searchable, PR-reviewed, AI-readable
ADRs, RFCs, per-feature specsdocs/adr/ docs/rfc/ docs/specs/Versioned with the code they govern
Backlog, PBIs, bugsThe work tracker (Azure Boards / Jira)Status, workflow, capacity live on the item

Working a tracker item -- where the context comes from

The most surprising row above: the backlog lives in Azure/Jira, not in docs/. So how do you get the spec when you pick up a PBI? The context isn't copied into the tracker -- it's linked by Req-ID back to the docs. The tracker holds the work; the docs hold the contract. Four things carry the context to you while you build:

1 - On the item itself

emit_backlog.py maps the story -> Description, the acceptance criteria -> its own field (never buried in Description), and the Source -> a trace ref -- the FRD/SRS ID it implements. The item says what it is and where it came from.

2 - The trace ref -> full context

Follow that FRD-TRK-001 through RTM.md to the behavior (FRD), constraints (SRS), design (SDD), and the decision (ADR). The deep context stays in one place; the item just points at it.

3 - /pbi-next assembles it

Returns the next ready PBI in full -- story, acceptance criteria, Definition of Done, the upstream IDs -- then reads those spec sections and summarizes the exact behavior to satisfy, so you start without re-reading everything. Routes by type: Story->dev, Bug->debugger, Spike->recon, Decision->architect.

4 - solution-recon = warm context

On an existing codebase, recon reads the real code first -- actual endpoints, field names, enum/status values, the pattern to mirror, FE-only vs backend-first. You build from spec plus verified code ground-truth, not assumptions.

And the link stays live both ways: emit_backlog.py writes the tracker IDs back into the RTM's Azure column, and the MCP server answers trace_requirement(...) / check_conformance() on demand -- so the item in Azure and the spec in the repo never drift apart.

Organized by scope & cadence

Artifacts are filed by how often they change and what scope they serve -- so fast-moving sprint work never pollutes durable knowledge. The four planes (the Wheelwright STANDARD.md):

PlaneWhatCadenceHome

The per-sprint folder holds the sprint's documents (plan, retro) -- not the work items. PBIs & bugs stay in the tracker (the system of record); the sprint folder references them by ID and freezes to a baseline at close. Per-sprint copies of items in the repo would duplicate the tracker -- exactly what the golden rule forbids.

Overhead? Bounded by your profile: lean stays flat (prd.md + RTM.md), mid adds per-feature specs, enterprise adds the full funnel + constitution. Folders exist only for what the profile populates, and closed sprints freeze to baselines instead of piling up -- so the scope ladder (product -> feature -> sprint -> item) costs nothing you don't use.

Not every agent writes a document

A fair question: do all 52 agents follow this writing-oriented approach? No -- and that's deliberate. Only the document agents produce durable markdown; the rest output code, reviews, analysis, or tracker items. What they share isn't a file format -- it's the contract each declares in its outputs / rtm_column. Output by type:

The unifying rule isn't "everyone writes a doc." Every agent (1) reads the artifact above it first -- ground, don't fabricate; (2) writes its output into one declared home; (3) links it by Req-ID through the RTM -- so even code and reviews trace back to a requirement. Durable / living docs carry the metadata header; transient outputs (reviews, standups, PR notes) are exempt by design.
