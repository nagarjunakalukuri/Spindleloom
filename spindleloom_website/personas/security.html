02

Driving a feature

the sequence, end to end
  1. Model before code. security-reviewer STRIDEs the SDD — assets, trust boundaries, top risks.
  2. Map to constraints. mitigations trace to SR-SEC ids so they're testable.
  3. Review the authz. explicitly, not incidentally — most breaches are authorization, not injection.
  4. Gate the release. findings carry severity + owner; on pass, the sign-off token feeds the release AND.

Pitfalls you avoid

  • Reviewing at the end instead of threat-modelling at design
  • A checklist pass that never looked at the authz design
  • Findings without severity or an owner

Troubleshooting

the gotchas the validators catch
end-only review
Security looked at only at the end.
Threat-model at design; late review finds expensive, structural problems.
skipped authz
A checklist pass that never read the authz design.
Review authorization explicitly — it's the most common real breach path.
ownerless finding
A risk with no severity or owner.
Every finding gets a severity and an owner, or it won't get fixed.
