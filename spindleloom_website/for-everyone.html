Spindleloom - how it works, end to end

Software, built like a wheel.

A spindleloom builds the wheel that keeps turning. Spindleloom is a crew of 52 specialist AI agents -- one for each step of building software, from the market question down to shipped, monitored code.

You don't open an app or learn 52 commands. You describe the work; the right specialist picks it up, does it, and hands the result to the next one along.

Say this to a teammate

The 30-second version

"It's a team of ~52 AI agents, each owning one step of building software -- market research, requirements, design, backlog, code review, releases. You don't learn a command per agent. You just describe what you want -- 'write a PRD for our login feature' -- and the right agent picks it up, does it, and hands off to the next."

-- the whole idea, in one breath

How you actually use it

Three moves, that's it

There's nothing to launch. The fleet loads with your project and waits for plain English. Every job is the same shape:

01

You describe it

Ask for an outcome in your own words. No agent names to memorize, no syntax.

# you type Write a PRD for a team expense tracker.
02

The right one acts

The matching specialist recognizes the job, reads the document above it for context, and produces the real artifact.

# prd-writer wakes up -> personas, user stories, acceptance criteria...
03

It hands off

The result feeds the next step, and discoveries push back upstream. The chain keeps the work joined up.

# next in line PRD -> backlog-manager -> sprint-planner...

The full run - idea to shipped

Eight stages, one turning wheel

This is the whole loop. You rarely run every agent -- pick the handful your team needs (ask doc-strategy-advisor). Each stage answers one question and hands the answer down.

working PLAN
1

Discover

Is there a market -- and why build it?

You get: a market case and clear, non-technical business goals the whole thing is justified against.

mrd-writerbrd-writer
2

Define

What should it do, for the person using it?

You get: product behavior (user stories), exact feature logic with edge cases, and measurable constraints -- speed, scale, security, accessibility.

prd-writerfrd-writersrs-writer
3

Design

What does it look like, and how is it built?

You get: screens and a design system, an architecture blueprint, the build spec, and the API + data contracts.

4

Decide

What did we choose, and why?

You get: proposals debated before building, decisions recorded so the reasoning survives, and the sanctioned tech every team aligns to.

architectrfc-facilitatoradr-writertech-radar-curator
5

Plan

What do we build first?

You get: a prioritized backlog of build-ready stories, sized by effort, fitted into a sprint you can actually finish.

backlog-managerestimation-facilitatorsprint-planner
6

Build

Build it -- and check it's right.

You get: code and tests, independently run-verified (maker/checker), then reviewed for quality, security, and accessibility before anything merges.

frontend-developerbackend-developertest-authorchange-verifiercode-reviewersecurity-revieweraccessibility-auditor
7

Ship

Get it out safely.

You get: automated delivery, a QA sign-off, an evidence-based go/no-go gate, and live monitoring once it's out.

pipeline-engineerqa-testerrelease-managersre
8

Learn

What did we learn -- and what's next?

You get: real usage metrics, a blameless retro with owned actions, and a tracked tech-debt register.

product-analyticsretrospective-facilitatortech-debt-keeper

-> Step 8 feeds straight back into step 5 -- the wheel turns. Holding it together: a Constitution (the standing rules every feature inherits) and the RTM -- one table that proves every requirement made it to a design and a test, so nothing is silently dropped.
